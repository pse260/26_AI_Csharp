# Horror
 - 호러게임의 개발하기 위한 방법 및 계획믈 Game Design Document로 출력하는 문서



## 키워드 - 게임 페일루나

### [Core Keywords: Lunatic Logic]
#### 1. 장르 및 분위기 키워드

* #Meta-Fiction (메타 픽션): 게임 속 세계와 현실의 경계가 무너지는 공포.
* #Text-Based_Horror (텍스트 기반 호러): 이미지보다 텍스트 묘사가 주는 상상력의 공포.
* #Liminal_Space (리미널 스페이스): 익숙하지만 어딘가 잘못된 느낌을 주는 공간.
* #Cryptic (불가해함): 논리적으로 설명되지 않는 기괴한 명령어와 전개.

#### 2. 핵심 게임플레이 키워드

* #Illogical_Progress (비논리적 진행):
* 보물을 버려야 길이 열림.
   * 상식 밖의 행동을 강요하여 유저의 도덕심이나 논리를 시험함.
* #Hidden_Coordinates (숨겨진 좌표):
* 게임 내 텍스트 속에 실제 위도/경도 또는 특정 주소를 숨김.
   * 현실 세계와의 연결고리 형성.
* #Corrupted_Command (오염된 명령어):
* 정상적인 시스템 메시지가 어느 순간부터 주인공의 목소리나 위협으로 변함.
* #Permadeath_Ambiguity (모호한 죽음):
* 게임 오버 시 단순 재시작이 아닌, 파일이 삭제되거나 메시지가 바뀌는 등의 영구적 변화.

#### 3. 내러티브 및 연출 키워드

* #Uncanny_Valley_of_Text (텍스트의 불쾌한 골짜기):
* 단조로운 문장 속에 갑자기 툭 튀어나오는 지나치게 상세한 고어 묘사.
* #Glitch_Aesthetic (글리치 미학):
* 고전 8비트 화면의 깨짐, 색상 반전, 의도적인 프레임 드랍.
* #White_Noise (백색 소음):
* 배경음악 대신 들리는 기분 나쁜 기계음과 정적의 교차.
* #Breaking_the_4th_Wall (제4의 벽 돌파):
* 플레이어의 PC 사용자 이름(OS 계정명)을 직접 언급하거나 파일을 생성함.

#### 4. 플레이어 경험(UX) 키워드

* #Paranoia (편집증): "누군가 나를 지켜보고 있다"는 느낌을 유발.
* #Frustration_as_Fear (공포로서의 좌절): 불친절한 시스템이 유저를 막다른 길로 몰아넣을 때 느껴지는 압박감.
* #Curiosity_Killed_the_Cat (호기심의 대가): 하지 말라는 행동을 했을 때 보상 대신 끔찍한 진실을 제공.

### 📄 [GDD] 텍스트 호러 게임 기획서
#### 1. 기본 개요 (General Information)

| 항목 | 내용 |
|---|---|
| 게임 제목 | (예: 심야의 도서관, 붉은 목소리 등) |
| 장르 | 텍스트 어드벤처, 심리적 호러, 미스터리 |
| 타겟 플랫폼 | PC (Steam/Itch.io), 모바일, 웹(Web) |
| 핵심 키워드 | 폐쇄 공포, 상상력 자극, 선택과 결과, 긴장감 |

#### 2. 스토리 및 세계관 (Story & World)

| 항목 | 내용 |
|---|---|
| 로그라인 | 한 문장으로 요약한 핵심 줄거리 (예: 기억을 잃고 깨어난 지하실에서 탈출해야 한다.) |
| 배경 설정 | 시간적/공간적 배경 (예: 1990년대 버려진 병원) |
| 주요 캐릭터 | 플레이어 캐릭터 및 조력자/적대자의 성격과 목적 |
| 핵심 갈등 | 플레이어를 위협하는 존재나 상황 (귀신, 살인마, 혹은 광기 등) |

#### 3. 게임플레이 메커니즘 (Gameplay Mechanics)

| 항목 | 내용 |
|---|---|
| 핵심 루프 | 텍스트 읽기 → 선택지 결정 → 결과 확인/상태 변화 → 스토리 진행 |
| 상태창(Status) | 체력(HP), 정신력(Sanity), 보유 아이템, 현재 시간 등 |
| 선택지 시스템 | 단순 분기형, 능력치 기반 선택, 시간 제한 선택지 등 |
| 인벤토리 | 획득한 단서나 도구를 확인하고 조합하는 기능 |

#### 4. 콘텐츠 및 시스템 (Content & System)

| 항목 | 내용 |
|---|---|
| 멀티 엔딩 | 배드 엔딩(다수), 노멀 엔딩, 진 엔딩의 조건 설정 |
| 저장 시스템 | 자동 저장(체크포인트) 또는 특정 구역에서만 저장 가능 |
| 사운드 효과 | 몰입감을 극대화할 효과음(발자국 소리, 숨소리) 및 배경음(BGM) |
| 시각 요소 | 단순 텍스트, 삽화(CG), 화면 흔들림 또는 글자 왜곡 연출 |

#### 5. 개발 단계별 체크리스트 (Development Roadmap)

| 단계 | 주요 작업 내용 |
|---|---|
| 프로토타입 | 핵심 스토리 시나리오 1장 작성 및 선택지 작동 확인 |
| 에셋 제작 | 스크립트 작성, 효과음 수집, UI 디자인 |
| 알파 테스트 | 전체 시나리오 연결 및 버그(무한 루프 등) 수정 |
| 폴리싱 | 텍스트 연출(타이핑 효과 등) 및 난이도 조절 |

### 💡 기획 팁:
#### 텍스트 호러는 직접 보여주지 않을 때 플레이어의 상상력이 더해져 가장 무섭습니다. 묘사는 구체적으로하되, 공포의 대상은 끝까지 신비감을 유지하는 것이 좋습니다.

### 📋 텍스트 호러 게임 제작 실행 계획

| 구분 | 실행 항목 | 세부 계획 및 방법 | 우선순위 |
|---|---|---|---|
| 1. 스토리 기획 | 시나리오 트리 구축 | 메인 플롯 설정 후, 선택지에 따른 분기점(Branch)을 마인드맵 도구로 시각화 | 최우선 |
| | 스크립트 작성 | 독자의 상상력을 자극하는 관용구, 소름 돋는 묘사 위주의 텍스트 집필 | 높음 |
| 2. 시스템 설계 | 스크립트 엔진 선택 | 'Ren'Py'(파이썬 기반)나 'Twine'(웹 기반) 등 텍스트 게임 특화 툴 선정 | 높음 |
| | 변수 시스템 설계 | 정신력(Sanity) 수치에 따라 텍스트가 다르게 보이거나(왜곡), 특정 선택지가 잠기는 로직 설계 | 보통 |
| 3. 연출 및 리소스 | 사운드 디자인 | 저작권 무료(CC0) 사이트에서 심장 박동, 쇠 긁는 소리 등 ASMR급 효과음 수집 | 높음 |
| | 텍스트 연출(VFX) | 글자가 깨지거나, 천천히 출력되거나, 갑자기 화면 전체를 덮는 등의 연출 스크립트 작성 | 보통 |
| 4. UI/UX 디자인 | 가독성 최적화 | 호러 분위기에 맞는 폰트 선정 및 눈이 피로하지 않은 배경색/글자색 조합 테스트 | 보통 |
| | 인터페이스 구성 | 인벤토리, 저장/불러오기, 로그(지난 대화 보기) 버튼 배치 | 낮음 |
| 5. 밸런싱 및 검수 | 데드락(Deadlock) 체크 | 특정 선택지 선택 시 스토리가 진행되지 않고 멈추는 구간이 없는지 전수 조사 | 높음 |
| | 공포 강도 조절 | 테스트 플레이를 통해 긴장감이 떨어지는 구간에 갑툭튀(Jump Scare)나 음향 효과 추가 | 보통 |
| 6. 배포 준비 | 빌드 및 테스트 | 윈도우, 맥, 모바일 등 타겟 플랫폼별 실행 파일 생성 및 호환성 체크 | 보통 |

#### 💡 실행을 위한 팁

   1. 스토리 보드 도구: Twine이나 Miro를 사용하면 복잡한 텍스트 분기를 한눈에 관리하기 좋습니다.
   2. 사운드의 중요성: 텍스트 게임에서 사운드는 그래픽 이상의 역할을 합니다. 보이지 않는 곳에서 들리는 소리로 공포를 유발하세요.
   3. 최소 기능 제품(MVP): 처음부터 너무 긴 시나리오보다는, 10분 내외의 단편 분량을 먼저 완성해 보는 것을 추천합니다.

| 4일차 | 테스트 및 마감 | 2h: 버그 수정 및 밸런스 조정 1h: UI/폰트 디자인 다듬기 1h: 최종 빌드 및 배포 | 완성도 제고 선택지 오류(무한 루프 등)를 전수 조사하고 최종 실행 파일을 생성합니다. |

## 게임 개발
